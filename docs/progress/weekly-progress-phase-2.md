## Interview Readiness

### Core Question: "Design a scraper for 100K URLs without getting banned"

**Cold Answer** (practiced, ~3 min):

"I'd approach this with three key constraints: concurrency limits, rate limiting, and graceful degradation.

First, **concurrency**: Use a Semaphore (Python) to limit simultaneous requests. I default to 5 concurrent because it's safe (most sites allow 5–10 req/s), observable (easy to debug), and efficient (low memory). At 5 req/s, 100K URLs = 6 hours.

Second, **rate limiting**: Parse HTTP headers (`Retry-After`, `X-RateLimit-*`) and implement exponential backoff with jitter. If a site returns 429, wait 2^N seconds + random(0,1s) before retrying. Max 3 retries.

Third, **graceful degradation**: When scraping fails (timeout, 503, parse error), log with context (source, attempt count, duration) and move on. Don't crash; don't get stuck.

For persistent scaling (1M+ URLs), I'd add:

- Residential proxies (Bright Data, Crawlbase) to rotate IPs
- Distributed scraping (Celery workers) with shared MongoDB backend
- Circuit breaker (if >30% fail over 5min, auto-disable source)
- Semantic caching: don't re-scrape unchanged content
"

**Interview signals demonstrated**:

- ✅ Constraints-first thinking (concurrency, rate, error handling)
- ✅ Trade-off awareness (5 vs 100 concurrent; why 5?)
- ✅ Practical implementation (Semaphore, backoff, logging)
- ✅ Scale awareness (what's next at 1M URLs)
- ✅ Testability (how you'd measure if it works)

### Follow-up #1: "How prevent a request ban?"

**Answer I practiced**:

"Three layers:

1. **Header rotation**: Vary User-Agent + Accept headers. Don't use identical headers for every request; bots tend to be boring/repetitive.

2. **Request pacing**: Wait between requests per-source (1–5s based on robots.txt crawl-delay). Use jitter (`random.uniform(1,5)`) so requests aren't perfectly spaced.

3. **Advanced (at scale)**: Residential proxy pool (IP rotation), detect anti-bot protections (Cloudflare, akamai) and fallback to API if available.

In my Phase 2 implementation, I focused on pacing + logging. Header rotation + proxies are enterprise features (Phase 3+ with budget).
"

**What I actually did**:

- Implemented timeout enforcement + exponential backoff
- Logged all requests with context (source, attempt, duration)
- Infrastructure for per-source concurrency limits (configurable)
- Note: Header rotation is future work (not in Phase 2, but documented in portfolio)

### Follow-up #2: "Concurrent vs sequential — trade-offs?"

**Answer I practiced**:

| N Workers | Time (100K) | Ban Risk | Memory |
| --------- | ----------- | -------- | ------ |
| 1         | 27 hours    | None     | ~100KB |
| 5         | 5.5 hours   | Low      | ~500KB |
| 20        | 1.5 hours   | Medium   | ~2MB   |
| 100       | 3 min       | High     | ~10MB  |
| ∞ (chaos) | Seconds     | 🚫 Banned| N/A    |

**My pick**: 5 concurrent, why:

- **Safe**: Most sites tolerate 5 req/s
- **Observable**: Easier to debug 5 failed requests than 100
- **Cost**: Cheap per-instance resource usage
- **Speed**: Still reasonable (6 hours for 100K)

**When to go higher**: Only with IP rotation + header variety (residential proxies).
"

## Learning Insights

### What Surprised Me

1. **uv dependency management**: Initially thought all scrapers (httpx, beautifulsoup, playwright) would be heavy. Turns out uv respects `--no-dev`, so production image is slim.

2. **Motor async client**: Coming from sync drivers (PyMongo), Motor's Semaphore-aware scheduling is transparent. MongoDB insert_many() returns immediately; no blocking threads.

3. **Fail-open complexity**: First instinct was strict (crash on MongoDB error). Realized this cascades: scraper down → ingestor down → whole pipeline down. Fail-open requires discipline: log everything, make errors observable.

### Mistakes Avoided

1. ❌ **Hardcoding Semaphore limits** → Now config-driven (`SEMAPHORE_LIMITS` dict)
2. ❌ **Swallowing exceptions** → Now logging all errors with context
3. ❌ **Only testing "happy path"** → Now testing 429/timeout/parser errors (70% coverage)

### Concept Clarity

Semaphore ≠ Rate Limiter

- Semaphore: Limit concurrency (max 5 requests in-flight)
- Rate Limiter: Limit request frequency (max 1 req per second)

**Semaphore(5) + 1s delay per request** ≠ Same thing. Semaphore is **concurrency**, delay is **rate limiting**. Both needed at scale.
